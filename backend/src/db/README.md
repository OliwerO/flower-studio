# `src/db/` — Postgres layer

Drizzle ORM + `pg` driver. Activated when `DATABASE_URL` is set; otherwise
the backend runs Airtable-only and these modules are inert.

## Files

| File | Purpose |
|---|---|
| `index.js` | Singleton `pg.Pool` + Drizzle handle. `connectPostgres()` runs at boot. |
| `schema.js` | Drizzle table definitions. Tables added per migration phase. |
| `migrate.js` | Standalone migration runner — `npm run db:migrate`. |
| `migrations/` | SQL migrations generated by `npm run db:generate`. Committed to git. |

## Workflow

1. Edit `schema.js` (add a column, table, etc.)
2. `npm run db:generate` — Drizzle diffs schema vs current migrations and
   writes a new `0NNN_*.sql` file in `migrations/`. Review the SQL.
3. Commit the migration with the schema change in the same PR.
4. `npm run db:migrate` — applies pending migrations to the DB at
   `DATABASE_URL`. Idempotent; run on every deploy.

## Why migrations are committed (not auto-generated on deploy)

Reviewable. A schema change is a deployable event — we want the SQL on
record, not regenerated each boot from a `.js` file (which can drift if
two devs land conflicting schema edits).
