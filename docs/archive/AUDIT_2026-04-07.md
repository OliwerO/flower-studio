## 1. Executive Summary

The app is **partially trustworthy** — reliable enough for daily operations but carrying several failure modes that silently corrupt data without alerting anyone. The dashboard's operational numbers (order counts, unpaid totals, today's revenue) are correct *most of the time*, but the margin for silent error grows every week without the fixes listed below.

**Can the dashboard be trusted?**  
> **With conditions.** Today's order count, delivery queue, and unpaid totals are sourced from a single `/dashboard` API call that is well-written. The Financial KPI tab, however, never auto-refreshes after writes — the owner can be staring at a 30-minute-old revenue figure without realizing it. Stock quantities are the least trustworthy: two races (the `/adjust` and `/write-off` routes both bypass the serialization queue) mean stems can vanish from the count silently every time two people adjust stock simultaneously.

**Confidence caveat — IMPORTANT:**  
Every finding in this report is labeled **code-only — production frequency unverified**. The Railway CLI was not installed and no production logs were read (user decision 2026-04-07). "This bug exists in the code" is not the same as "this bug fired in production today." The owner must treat severity ratings as *potential* severity — what would happen if the bad path triggered — not as confirmed production incidents.

**One important correction from the pre-audit plan:**  
The plan listed "bouquet edit does not deduct stock for new flowers" as CRITICAL. After reading `orderService.js:340–354` line-by-line, this bug **appears to have already been fixed** — the code does call `atomicStockAdjust` for new lines. This is noted under CORRECTED FINDINGS. ## 3. CRITICAL Findings

### C1 — PO Evaluation retry causes double stock addition
**File:** `backend/src/routes/stockOrders.js:382–462`  
**How confirmed:** Direct Read of the evaluate route.

**What happens (IE analogy):** During PO evaluation, the florist marks each batch as Accepted. The system calls `receiveIntoStock`, then creates an audit record in `STOCK_PURCHASES`. These are two separate Airtable writes inside a per-line try/catch. If `receiveIntoStock` succeeds (stock is incremented) but the `STOCK_PURCHASES.create` at line 405–412 throws a network error, the inner catch at line 458 fires. The line is **not** marked `PROCESSED`. The PO is flagged `EVAL_ERROR`. When the owner clicks Retry, the idempotency check at line 387 (`if (line['Eval Status'] === PO_LINE_STATUS.PROCESSED)`) passes right through — and `receiveIntoStock` runs again, crediting stock a second time.

**Plan correction:** The pre-audit plan said "the line is still flagged PROCESSED before side-effects." This is wrong. PROCESSED is correctly set at the end (lines 451–455). The real vulnerability is that partial success within `receiveIntoStock` itself is not idempotent. The mechanism is: retry-on-error + partial success = double credit.

**Blast radius:** Stock count for that flower inflated by accepted quantity. All future orders drawn from that stock item can oversell. Dashboard will show higher available quantity than physically exists. Loss invisible until physical count.

**Recommended fix:** After `receiveIntoStock` succeeds, immediately write a per-step completion flag (e.g., `Receive Status: ok`) to `STOCK_ORDER_LINES` before creating the purchase record. Check that flag on retry. Alternatively, make `receiveIntoStock` check if a `STOCK_PURCHASES` record for this PO+line already exists before writing.

**SQL migration relevance:** ✅ YES — This is one of the primary arguments for moving to PostgreSQL. A real transaction would wrap receiveIntoStock + create(STOCK_PURCHASES) atomically. Either both succeed or neither does. Airtable has no transactions.

### C2 — createOrder rollback can fail silently, leaving orphan records
**File:** `backend/src/services/orderService.js:161–183`  
**How confirmed:** Direct Read.

**What happens:** `createOrder` has a rollback block in its catch clause. If the rollback itself fails (e.g., Airtable rate-limit during high load), errors are pushed to `rollbackErrors` at line 168 and only `console.error`'d at line 179. The original error rethrows, the user sees "Order creation failed. Partial records have been cleaned up." — but they may not have been. An ORDER record (or ORDER_LINE records) can persist in Airtable with no order visible in the app.

**Blast radius:** Orphan order records accumulate in Airtable. Stock deductions that happened before the failure point are reversed (when rollback works), but if reversal fails, stock is also phantom-deducted. User sees a failure toast but the Airtable base has garbage. Low frequency (requires Airtable failure during rollback), but undetectable without manual base inspection.

**Recommended fix:** Write a dedicated `ROLLBACK_ERRORS` log entry to Airtable (a simple table: orderId, timestamp, error, affected records) whenever rollback encounters failures. This creates a permanent audit trail the owner can review instead of relying on Railway stdout, which scrolls away.

**SQL migration relevance:** ✅ YES — PostgreSQL transactions make this entire rollback block unnecessary. `BEGIN; ... ROLLBACK;` is atomic.

## 4. HIGH Findings

### H1 — /adjust AND /write-off both bypass the stock serialization queue
**File:** `backend/src/routes/stock.js:230–254` (adjust), `257–318` (write-off)  
**How confirmed:** Direct Read.

**What happens:** `atomicStockAdjust` in `airtable.js:107–124` is the correctly-serialized path — it queues through `stockQueue` (concurrency 1), reads current quantity fresh, adds delta, writes back. But `POST /api/stock/:id/adjust` (line 238) and `POST /api/stock/:id/write-off` (line 267) both do a raw `db.getById` → compute → `db.update` **outside** `stockQueue`. If two florists both open the stock adjustment screen and tap at the same moment, they both read the same quantity. One adjustment is lost.

**Plan correction:** The plan only mentioned `/adjust`. After reading the file, `/write-off` has the same race. Both need fixing.

**Blast radius:** Every concurrent stock adjustment silently loses stems. This compounds daily in a shop with 2–4 florists. The quantity drift is invisible — nobody sees an error.

**Recommended fix:** In both routes, replace the manual read → compute → write pattern with `db.atomicStockAdjust(req.params.id, delta)`. The delta for write-off is `-quantity`. This is a 2-line change per route.

**SQL migration relevance:** ✅ YES — `UPDATE stock SET qty = qty - $delta WHERE id = $id` is inherently atomic in SQL. No race possible.

### H2 — Order ↔ Delivery cascade can desync (both directions)
**File:** `backend/src/services/orderService.js:224` (Order→Delivery), `backend/src/routes/deliveries.js:112–120` (Delivery→Order)  
**How confirmed:** Direct Read.

**What happens:**

*Direction 1 — Order→Delivery (via `transitionStatus`):*  
Line 224: `await db.update(TABLES.DELIVERIES, deliveryId, deliveryPatch).catch(() => {});`  
If the delivery status update fails, the failure is silently swallowed. The Order record says "Out for Delivery"; the Delivery record still says "Ready."

*Direction 2 — Delivery→Order (via `deliveries.js` PATCH):*  
Line 114 updates the linked Order status. Line 120 updates the Delivery record itself. If line 120 throws after line 114 succeeded, the Order is updated but the Delivery is not — permanent desync.

**Plan correction:** The plan said both use `.catch(() => {})`. Only the Order→Delivery direction does. The Delivery→Order direction has no `.catch` but still has a sequencing risk.

**Blast radius:** Driver marks delivery "Out for Delivery." Florist still sees "Ready" and may prep another call to the customer. Or vice versa. Operational confusion, possible duplicate customer contact.

**Recommended fix (Order→Delivery):** Remove the `.catch(() => {})` — let the error propagate, return 500 to the client, and show the user an error toast. Silencing it is worse than a visible failure.  
**Recommended fix (Delivery→Order):** Move the Order update to happen *after* the Delivery update, so if the Delivery update fails, neither record is changed.

### H3 — Bouquet edit silently swallows failed line deletions
**File:** `backend/src/services/orderService.js:312–314`  
**How confirmed:** Direct Read.

```javascript
if (rem.lineId) {
  await db.deleteRecord(TABLES.ORDER_LINES, rem.lineId).catch(() => {});
}
```

**What happens:** When a florist removes a flower from a bouquet, the ORDER_LINE record in Airtable may silently fail to delete. The API returns 200 ("bouquet updated"), the frontend refreshes, and the deleted line reappears — because it was never actually removed. Stock adjustment for the removal may also be out of sync.

**Blast radius:** Phantom order lines inflate bouquet cost/sell totals. Stock adjustment happens (return or write-off) but the line still exists, creating a phantom stem on the book.

**Recommended fix:** Remove `.catch(() => {})`. Let the error propagate and return a 500. The frontend already handles this (it shows `err.response?.data?.error` as a toast).

---

### H4 — Settings config writes are fire-and-forget
**File:** `backend/src/routes/settings.js:386–389`  
**How confirmed:** Direct Read — confirmed by comment "fire-and-forget" at line 386.

**What happens:** When the owner updates a config value (delivery fee, lot sizes, etc.), the in-memory `config` object is updated immediately and `200 OK` is returned. Then `saveConfig()` is called with `.catch`. If the Airtable write fails, the owner sees success — but the next Railway restart restores the old config from Airtable, silently reverting the change.

**Blast radius:** Owner changes the default delivery fee from 20 to 30 zł. Railway restarts overnight (common on the free tier). Next morning, all new orders have the old delivery fee. Owner doesn't notice until a customer complains.

**Recommended fix:** Either `await saveConfig()` before responding (adds ~200ms, acceptable for a settings write), or explicitly show a warning toast if the background save fails (requires an SSE notification or a second fetch).

---

### H5 — driver-of-day partial-loop with pre-committed in-memory state
**File:** `backend/src/routes/settings.js:343–366`  
**How confirmed:** Direct Read.

**What happens:** Line 346 sets `daily.driverOfDay = driverName` **before** the assignment loop runs. If any delivery update in the loop throws (line 357), `next(err)` returns a 500. The owner sees an error toast. But `daily.driverOfDay` is already set — future orders will auto-assign to the chosen driver, even though some of today's existing deliveries were not assigned. Partial assignment is confirmed, partial assignment is invisible (the 500 tells the owner "something went wrong" but not what was and wasn't assigned).

**Blast radius:** Some of today's deliveries assigned to driver, others still unassigned. The "unassigned deliveries" alert on the dashboard may not fire because `driverOfDay` is set, masking the problem.

**Recommended fix:** Set `daily.driverOfDay` after the loop completes successfully, or explicitly report which deliveries succeeded/failed in the response so the frontend can show a specific warning.

## 7. SQL Migration Verdict

**Recommendation: Plan the migration, but fix H1 (race conditions) and C1 (PO evaluation retry) first.**

The case for moving to PostgreSQL rests primarily on **Findings C1, C2, H1, and M6** — not on performance. Here is what the SQL move would and would NOT solve:

### What SQL would fix (structural improvements):

| Finding | How SQL fixes it |
|---------|-----------------|
| C1 — PO evaluate double-stock-addition | `BEGIN; UPDATE stock; INSERT purchases; UPDATE line_status; COMMIT;` is atomic. No retry-after-partial-success problem. |
| C2 — createOrder rollback failures | `BEGIN; ... ROLLBACK;` — no manual rollback block needed. Either everything commits or nothing does. |
| H1 — /adjust and /write-off races | `UPDATE stock SET qty = qty + $delta WHERE id = $id` is atomic row-level operation. No read-then-write race possible. |
| M6 — No 429 retry | SQL never rate-limits its own operations. The 5-req/sec constraint disappears. |
| M7 — Slow sequential writes | Batch inserts, transactions, and prepared statements all compress the latency. |

### What SQL would NOT fix:

- **H2 (cascade desync)** — the two-record desync is a logic problem, not a storage problem. SQL with two UPDATE statements in a transaction would fix it, but you still need to write the transaction. It's not automatic.
- **H3 (silent line deletion)** — this is a `.catch(() => {})` code smell. Needs to be removed regardless of DB.
- **H4 (fire-and-forget config)** — depends on whether settings are in-memory or in DB. If moved to SQL, a failed `UPDATE` would be synchronous and visible. But the architectural choice (in-memory config) needs to change too.
- **H5 (partial driver assignment)** — logic problem, not a storage problem.
- **M1 (SSE no catch-up)** — frontend architecture, independent of DB.

### Migration timeline suggestion:

1. **This sprint:** Fix H1 (race conditions) — 10-line change, immediate risk reduction.
2. **Next sprint:** Fix C1 (PO evaluation idempotency) — medium effort, high value.
3. **After stable:** Begin SQL schema design. PostgreSQL on Railway is already available in the stack.
4. **Migration order:** Stock first (high-frequency race conditions), then Orders (most complex), then Customers/Config last.
