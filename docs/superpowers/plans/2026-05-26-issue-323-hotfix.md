## Root cause (confirmed)

`receiveIntoStock` does not propagate Variety attrs (`type_name`, `colour`, `size_cm`, `cultivar`) from the PO line to the new dated Batch. Sibling site at `stockPurchases.js:33` has the same omission. Orig Demand Entries with `type_name=NULL` (legacy / pre-Y-model rows) stay NULL forever.

Single bug, three downstream symptoms:
1. New Batch invisible in `/stock?grouped=true` (Y-model grouped view).
2. FEFO router (#322) cannot compute Variety key for the new Batch → orders route past it.
3. Legacy picker (current prod path under flag-off) prefers undated DE by name match → orders pile onto orig.

## Hotfix scope (per PRD #324 line 150)

> "receiveIntoStock Variety propagation (precondition, hotfix scope). Variety attrs (type_name, colour, size_cm, cultivar) flow from the PO line to the new Batch at receipt time. Same fix applied to `stockPurchases.js:33` (sibling absorption site). Orig DE auto-backfilled with the same attrs when its `type_name` is currently NULL. This is the issue-#323 hotfix; it lands ahead of the rest of this PRD as its own PR but is captured here for completeness because the trace surface only works when Variety attrs are present."

### Files to touch

- `backend/src/routes/stockOrders.js` lines 577-619 (`receiveIntoStock` function)
  - Accept Variety attrs as additional function parameters (Type, Colour, Size, Cultivar from the PO line context).
  - Copy them into the `stockRepo.create({...})` call for the new dated Batch.
  - If the orig `stockItem['Type']` is null and the PO line carries non-null attrs, include them in the `stockRepo.update(stockItemId, {...})` call further down (the price + last_restocked update) so the orig DE gets backfilled in the same transaction.
- `backend/src/routes/stockOrders.js` call sites of `receiveIntoStock` (lines 781, 830). Pass the PO line's Variety attrs through.
- `backend/src/routes/stockPurchases.js:33` — apply the same pattern. Audit the function signature; this site does not go through `receiveIntoStock` but mirrors its absorption math. Either: (1) refactor both to call a shared helper, or (2) duplicate the attr-propagation fix in both (smaller diff, less coupling).

### Pure additions (no semantic change to absorption math)

- Absorption math (`batchQty = qty + existingQty` when negative) stays. Per ADR-0002 design intent. NOT in scope to change.
